# TradePost Pro - 自動スケーリング設定ガイド

本ドキュメントでは、TradePost Proのユーザー増加やトラフィック急増に対応するための、自動スケーリング（Auto Scaling）と高可用性（HA）インフラの構築手順について解説します。

## 1. スケーリング戦略の概要

TradePost Proは、以下のような特性を持つシステムです。

- **朝7時のスパイクアクセス**: スケジューラーによる一斉投稿処理が集中
- **画像・動画生成の負荷**: CPU/メモリを大量に消費する処理
- **外部APIへの依存**: 各SNS APIのレート制限とレスポンス待ち

これらの特性に対応するため、以下の3層構造でのスケーリングを推奨します。

### スケーリングフェーズ

| フェーズ | ユーザー数目安 | インフラ構成 | 推定月額コスト |
|---------|-------------|-------------|-------------|
| **フェーズ1（初期）** | 1〜100人 | シングルVPS (ConoHa 2GB) + Xserver | 約5,000円 |
| **フェーズ2（成長期）** | 100〜500人 | VPS (ConoHa 4GB) + マネージドDB | 約15,000円 |
| **フェーズ3（拡大期）** | 500〜2000人 | ロードバランサー + 複数APIサーバー + Redis + DB | 約40,000円 |
| **フェーズ4（大規模）** | 2000人以上 | AWS (ALB + ECS/EKS + RDS + ElastiCache) | 10万円〜 |

## 2. フェーズ3（拡大期）のインフラ構築手順

ユーザー数が500人を超え、単一のVPSでは処理が追いつかなくなった場合の「スケールアウト（台数追加）」の構成です。

### 2.1. システムアーキテクチャ図

```text
[ユーザー / クライアント]
       │
       ▼
[ロードバランサー (Nginx / HAProxy)]
       │
       ├──► [APIサーバー 1 (FastAPI)] ──┐
       ├──► [APIサーバー 2 (FastAPI)] ──┼──► [マネージドデータベース (MySQL)]
       └──► [APIサーバー N (FastAPI)] ──┤
                                        │
                                        └──► [Redis (キュー・キャッシュ)]
                                                  │
                                                  ▼
                                         [ワーカーサーバー (Celery)]
                                         (画像生成・SNS投稿専用)
```

### 2.2. コンポーネントの分離

**1. データベースの分離**
ローカルのSQLite/MySQLから、ConoHaの「データベースサーバー」またはAWSの「RDS」へ移行します。

**2. ジョブキュー（Redis）の導入**
朝7時の投稿処理を分散させるため、現在のメモリ内キューからRedisベースの分散キュー（Celery等）へ移行します。

**3. ワーカーサーバーの独立**
画像・動画生成などCPUを消費する処理は、APIサーバーから切り離し、専用の「ワーカーサーバー」で非同期処理します。

## 3. Docker Composeを使ったスケーリング（1台の強力なサーバー内）

AWS等へ移行する前に、まずは1台の強力なサーバー内でDockerコンテナを増やす方法です。

`docker-compose.yml`のワーカー設定：

```yaml
services:
  # ... (その他のサービス)

  worker:
    build:
      context: .
      dockerfile: docker/Dockerfile.backend
    command: celery -A backend.worker worker --loglevel=info
    environment:
      - REDIS_URL=redis://redis:6379/0
      - DATABASE_URL=mysql+pymysql://user:pass@db:3306/tradepost
    depends_on:
      - redis
      - db
    deploy:
      replicas: 3  # ワーカーコンテナを3つ起動
      resources:
        limits:
          cpus: '1.0'
          memory: 1G
```

スケールアップのコマンド：
```bash
docker-compose up -d --scale worker=5
```

## 4. AWSへの移行（本格的なAuto Scaling）

数千人規模のユーザーを抱える場合、AWSのマネージドサービスを活用した完全な自動スケーリング環境への移行を推奨します。

### 4.1. AWSサービスの選定

- **コンピューティング**: Amazon ECS (Fargate)
- **データベース**: Amazon RDS for MySQL (Multi-AZ)
- **キャッシュ/キュー**: Amazon ElastiCache for Redis
- **ロードバランサー**: Application Load Balancer (ALB)
- **ストレージ**: Amazon S3 (ConoHaオブジェクトストレージから移行)

### 4.2. Auto Scaling グループの設定

ECS Fargateを使用する場合、CPUやメモリの使用率に応じて自動的にコンテナ（タスク）を増減させることができます。

**スケーリングポリシーの推奨設定：**

1. **ターゲット追跡スケーリング（Target Tracking）**
   - **CPU使用率**: 70%を維持するように設定
   - **メモリ使用率**: 80%を維持するように設定

2. **スケジュールに基づくスケーリング（Scheduled Scaling）**
   朝7時の投稿スパイクに備え、事前にサーバー台数を増やします。
   - **6:45**: 最小タスク数を1から5へ変更（スケールアウト）
   - **8:00**: 最小タスク数を5から1へ戻す（スケールイン）

### 4.3. SQS/Celeryキューに基づくスケーリング

APIサーバーのCPU使用率ではなく、「未処理の投稿ジョブ数」に応じてワーカーサーバーを増やすのが最も効率的です。

- **CloudWatch アラーム**: `Redisのキュー長 > 100` でアラーム発火
- **アクション**: ワーカータスクを+2追加

## 5. ボトルネック対策とチューニング

スケーリングを行っても解決しない「アプリケーション層のボトルネック」への対策です。

### 5.1. データベースの最適化
- **インデックスの追加**: ユーザーIDや日付での検索が多いため、適切なインデックスを付与
- **コネクションプーリング**: SQLAlchemyの`pool_size`と`max_overflow`を調整
- **リードレプリカ**: 参照クエリ（ダッシュボード表示など）をリードレプリカに逃がす

### 5.2. API制限の回避（SNS連携）
- 各SNS APIには厳しいレート制限（例：Xは15分に50回など）があります。
- サーバーを増やしても、同じAPIキーや同じIPからのリクエストは制限されます。
- **対策**: ユーザーごとのAPIキー使用、リクエストの分散（ジッターの導入）、エラー時の指数バックオフ（Exponential Backoff）の実装。

### 5.3. 画像・動画生成の最適化
- PillowやMoviePyの処理は重いため、可能であればAWS Lambdaや専用のGPUインスタンスに処理をオフロードすることを検討してください。

---

## 6. 監視とアラート

自動スケーリング環境では、サーバーが勝手に増減するため、適切な監視が不可欠です。

- **Datadog / Mackerel**: インフラ全体の統合監視
- **Sentry / New Relic**: アプリケーションパフォーマンス監視（APM）
- **Slack / LINE Notify**: スケールアウト・スケールイン発生時の通知

## 7. まとめ

TradePost Proのインフラは、初期はコストを抑えたVPS構成でスタートし、ユーザーの増加と収益の拡大に合わせて、段階的にRedisの導入、データベースの分離、そして最終的にはAWS等での完全なAuto Scaling環境へと進化させることを推奨します。
